# Core Business Flows

## 1. Роли в системе

### Обычный пользователь

Пользователь по умолчанию:

- попадает в приложение через Telegram Mini App или Telegram widget
- получает свой user profile
- может читать новости
- может открывать релизы
- может покупать релизы целиком и отдельные треки
- может лайкать, писать комментарии, ставить реакции
- может подписываться на людей
- может пополнять внутренний баланс
- может привязать TON-кошелек
- может минтить NFT только для тех релизов, которые куплены целиком и разрешены к mint

### Кандидат в артисты

Обычный пользователь может подать заявку на артистский статус. Для этого он заполняет анкету, где указывает:

- display name артиста
- bio
- avatar / cover
- TON wallet address
- reference links
- примечание для модерации

После подачи заявки:

- в конфиге создается `artistApplication`
- админам уходит уведомление
- пользователь ждет решения

### Артист

После одобрения заявки:

- создается отдельный `artistProfile`
- артист получает доступ к `Студии`
- может редактировать публичный профиль артиста
- может создавать релизы
- может управлять монетизацией
- может запрашивать вывод средств

Важно:

- `user profile` и `artist profile` должны восприниматься как две разные сущности
- пользовательский профиль нужен для социальной идентичности человека
- артистский профиль нужен для публикации и продажи контента

### Администратор

Администратор:

- модерирует заявки артистов
- модерирует выплаты
- управляет каталогом, промо, настройками, витринами
- контролирует заказы и контент

## 2. Главный сценарий обычного пользователя

### Вход в приложение

1. Пользователь открывает Mini App.
2. Приложение поднимает Telegram session.
3. Создается или подтягивается user identity.
4. Пользователь попадает в `Новости`.

### Исследование контента

Пользователь может:

- перейти в `Новости`
- перейти в `Релизы`
- открыть страницу артиста
- открыть внешний профиль пользователя
- открыть конкретный релиз

### Покупка релиза

Текущая целевая продуктовая модель:

- релиз состоит из нескольких треков
- у релиза есть несколько форматов
- у каждого трека может быть своя цена
- релиз можно купить целиком в выбранном формате
- релиз можно купить повторно только в другом формате
- треки можно покупать поштучно
- NFT минтится только на весь релиз, а не на отдельный трек
- не каждый релиз обязан быть mintable

### Что попадает в коллекцию пользователя

В коллекции должны отображаться:

- полностью купленные релизы
- частично купленные релизы
- прогресс по трекам внутри релиза
- наличие NFT-апгрейда

Это важно, потому что иначе покупка отдельных треков визуально теряется.

## 3. Социальный сценарий

Пользователь может:

- подписываться на других людей
- смотреть подписчиков и подписки
- комментировать посты
- комментировать релизы
- ставить реакции
- делиться профилем
- копировать username

Логика социальной части сейчас смешивает два типа сущностей:

- обычные публичные профили пользователей
- публичные профили артистов

Это допустимо как UI-паттерн, но на уровне данных их нужно держать отдельно.

## 4. Сценарий превращения пользователя в артиста

### Целевой флоу

1. Пользователь открывает настройки.
2. Выбирает путь `Стать артистом`.
3. Заполняет анкету.
4. Анкета попадает в админку.
5. Админ:
   - одобряет
   - отклоняет
   - или переводит в `needs_info`
6. Пользователь получает уведомление.
7. При одобрении появляется artist profile и доступ к студии.

### Текущее состояние

Это уже частично реализовано:

- есть `artistApplication`
- есть admin moderation route
- есть уведомления
- есть отдельный `/studio`

Но продуктово еще важно закрепить правило:

- пользовательский профиль и профиль артиста не должны визуально сливаться в один и тот же экран

## 5. Сценарий артиста

### Вкладка `Студия` внутри профиля

Это должен быть краткий dashboard:

- сколько релизов
- сколько опубликовано
- сколько продаж
- сколько реакций и комментариев
- сколько донатов и активных подписок
- сколько доступно к выводу

### Отдельная `/studio`

Это уже admin-зона артиста. Здесь он:

- редактирует artist profile
- создает релизы
- управляет будущими релизами
- отправляет запрос на вывод средств

## 6. Сценарий пополнения баланса

Пользователь может пополнять внутренний баланс:

- через Telegram Stars
- через TON

Баланс используется как внутренняя покупательная способность внутри приложения и как источник списания gas fee для sponsored NFT mint.

## 7. Сценарий NFT

### Что должно быть правилом продукта

- NFT не заменяет покупку релиза
- NFT является улучшением уже купленного релиза
- NFT минтится только для релиза, купленного целиком
- NFT не минтится для одного трека
- NFT доступен только для релизов с `isMintable = true`

### Что получает пользователь

- on-chain ownership
- запись о mint внутри social state
- улучшенный статус релиза в профиле и ленте

## 8. Сценарий администратора

Администратор управляет:

- artist applications
- artist payouts
- каталогом
- промо
- настройками
- блогом
- витринами
- заказами

Это уже полноценный operational backoffice.

## 9. Ключевые продуктовые инварианты

Их важно не ломать при дальнейшем развитии:

- пользователь создается автоматически при входе
- артист не рождается автоматически, а только через заявку и approval
- релиз можно покупать по трекам и целиком
- NFT можно минтить только после полной покупки релиза
- вывод артисту идет только на TON-кошелек
- payout нельзя делать мгновенно без hold и admin review
- tab bar показывается только на главных экранах

## 10. Нерешенные продуктовые вопросы

- Нужно ли держать отдельный экран `Профиль артиста`, если уже есть внешний профиль пользователя.
- Нужно ли объединять `artist page` и `public profile` в одну модель.
- Какая точная бизнес-норма hold на выплаты: `21 день` или другая величина.
- Нужно ли в будущем делать автоматический payout в TON после approve.
- Нужно ли разделить покупки на `ownership`, `license`, `collector upgrade`.
